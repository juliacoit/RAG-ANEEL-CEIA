SLIDE 2 — O PROBLEMA QUE RESOLVEMOS
================================================================================

TÍTULO: Legislação que ninguém consegue consultar rápido

COLUNA ESQUERDA — Contexto:
  • A ANEEL publica centenas de atos normativos por ano
  • Resoluções, despachos e portarias em PDFs, HTMLs e planilhas
  • Pesquisa manual é lenta, fragmentada e propensa a erro
  • Nenhuma ferramenta de busca em linguagem natural disponível

COLUNA DIREITA — Nossa solução:
  Sistema RAG que responde qualquer pergunta sobre legislação elétrica
  com citação direta da fonte — sem alucinações

NÚMEROS DE IMPACTO (grandes, centralizados):
  562.152 chunks indexados · 25.202 arquivos processados · 3 anos de legislação

SLIDE 4 — P1: ENGENHARIA DE DADOS
================================================================================

TÍTULO: P1 — Data Engineer · Ingestão, Parse e Chunking

TRÊS COLUNAS:

  ① DOWNLOAD (curl_cffi)
    • Contorna Cloudflare via impersonação TLS Chrome 124
    • 25.202 arquivos baixados com sucesso
    • Formatos: PDF · HTML · XLSX · ZIP · RAR
    • Download paralelo com workers configuráveis
    • Falhas registradas para reprocessamento

  ② PARSING (PyMuPDF · pdfplumber · BeautifulSoup4 · openpyxl)
    • PDFs digitais → PyMuPDF (texto limpo, detecção de tachado)
    • PDFs com tabelas → pdfplumber (estrutura de células preservada)
    • HTMLs → BeautifulSoup4
    • Planilhas XLSX/XLSM → openpyxl linha por linha
    • Cabeçalho injetado nos chunks de dados das planilhas

  ③ CHUNKING (LangChain RecursiveCharacterTextSplitter)
    • 800 chars por chunk · 120 de overlap
    • Chunks < 50 chars descartados
    • Metadados por chunk: número, ano, tipo, título, fonte, chunk_index

DADO DE DESTAQUE:
  578.319 chunks gerados (16.167 ementas + 562.152 PDFs/HTMLs/XLSXs)

SLIDE 5 — P2: BUSCA HÍBRIDA
================================================================================

TÍTULO: P2 — Search Engineer · Indexação e Recuperação

DIAGRAMA CENTRAL (duas colunas que se fundem):

  BUSCA SEMÂNTICA                    BUSCA LEXICAL
  ─────────────────                  ───────────────────
  Modelo: MiniLM-L12-v2 (local)      Algoritmo: BM25Okapi
  Dimensões: 384                     Corpus: 562.152 documentos
  Vector Store: Qdrant               Tokenização em PT-BR
  562.152 vetores                    Serializado em pickle
  Distância cosseno                  Score TF-IDF ponderado
           ↘                                  ↙
               WEIGHTED HYBRID SCORE
               0.6 × semântico + 0.4 × BM25

FUNCIONALIDADES AVANÇADAS:
  • HyDE — embedding sobre trecho hipotético (não sobre a query pura)
  • Expansão de contexto — chunks vizinhos (±2 por chunk base)
  • Filtros automáticos — número e ano do ato extraídos da query por regex
  • Fallback sem filtros — busca "modo menção" quando score < 0.5
  • Pool de 4 conexões Qdrant — scrolls paralelos para menor latência

SLIDE 6 — P3: API, LLM E INTERFACE
================================================================================

TÍTULO: P3 — LLM Engineer · API, Geração e Avaliação

QUATRO COLUNAS:

  ① API (FastAPI)
    • POST /query — pergunta → resposta com citação
    • GET /health — monitoramento
    • GET /docs   — Swagger automático
    • Cache LRU: 100 entradas, TTL 5min
    • ThreadPoolExecutor (paralelismo)
    • Timeout 12s no otimizador

  ② Query Optimizer (Groq)
    • Rewriting com vocabulário ANEEL
    • HyDE (trecho hipotético)
    • Decomposição em sub-queries
    • Memória: últimos 6 turnos
    • Filtros automáticos (número/ano)
    • Detecção de mudança de contexto
    • Roteamento analítico (pandas, sem LLM)

  ③ LLMs com Fallback
    Principal: Groq llama-3.3-70b
    • 100k tokens/dia gratuito
    Fallback automático: DeepSeek R1
    • Ativo quando Groq retorna 429
    • API OpenAI-compatível
    6 prompts adaptativos:
    BUSCA · RESUMO · COMPARACAO
    TABELA · ESPECIFICA · MENCAO

  ④ Interface (Streamlit)
    • Histórico: 10 turnos
    • Scores semântico + BM25
    • Indicador de cache hit
    • Tipo de resposta detectado
    • Roteamento analítico direto

FRASE DE DESTAQUE:
  "Sistema responde com citação direta da fonte. Zero inferência externa."

SLIDE 7 — PROBLEMAS RESOLVIDOS NA PRÁTICA
================================================================================

TÍTULO: Desafios Técnicos Superados

TRÊS CARDS (com problema → solução → resultado):

  CARD 1: Planilhas XLSX sem contexto
  ────────────────────────────────────
  Problema: chunks de XLSX chegavam ao LLM sem saber o que cada coluna
            representava (ex: "0.03 | 0.025 | 0.025" sem cabeçalho)
  Solução:  injeção automática do cabeçalho do chunk 0 nos chunks filhos
            + boost de +0.3 no score XLSX quando tipo = "tabela"
  Resultado: "Qual a taxa de depreciação para TURBINA HIDRÁULICA?"
             → "2,5% ao ano" — correto, com citação e unidade ✓

  CARD 2: Documentos não disponíveis na base
  ──────────────────────────────────────────
  Problema: quando um ato não estava indexado, o sistema retornava
            "Não encontrado" sem aproveitar menções em outros documentos
  Solução:  fallback sem filtros (score < 0.5) → busca chunks que
            MENCIONAM o ato solicitado → prompt MENCAO no LLM
  Resultado: "O que estabelece a REN 687?"
             → "Não disponível na base, mas é mencionada na REN 956/2021
                e REN 1000/2021 no contexto de microgeração distribuída." ✓

  CARD 3: Latência de 16–32 segundos
  ────────────────────────────────────
  Problema: busca levando 16–32s no Windows (CPU, scrolls sequenciais)
  Soluções: cache de embeddings (200 entradas), pool de 4 conexões Qdrant,
            ThreadPoolExecutor para scrolls paralelos, expansão de vizinhos
            desativada para tipos que não precisam
  Resultado: busca tipo tabela (cache quente): ~2,3s ✓
