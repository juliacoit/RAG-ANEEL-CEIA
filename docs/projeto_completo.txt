1. VOCÊ FEZ O QUÊ EXATAMENTE?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ✅ Download dos PDFs (e outros formatos)
     — ~25.202 arquivos baixados da ANEEL: PDFs, HTMLs, XLSXs, RARs
     — Contorno do Cloudflare via curl_cffi (impersonação de Chrome 124)

  ✅ Parsing / Extração de texto
     — PDFs digitais: PyMuPDF (fitz) para texto, tachado e detecção de imagens
     — PDFs com tabelas: pdfplumber para preservar estrutura
     — PDFs escaneados (imagem): marcados como "pdf_imagem" (sem OCR por texto)
     — HTMLs: BeautifulSoup4
     — XLSXs / XLSMs: openpyxl
     — RARs e ZIPs: extração com rarfile e zipfile

  ✅ Chunking
     — Ementas (JSONs): RecursiveCharacterTextSplitter (LangChain)
     — PDFs/HTMLs/XLSXs: chunking por parágrafo/bloco com overlap configurável
     — Metadados por chunk: numero, ano, tipo_codigo, titulo, fonte_texto, chunk_idx

  ✅ Vetorização (Embeddings)
     — Modelo: sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2 (384 dims)
     — Rodando localmente, sem chamada de API externa
     — Em migração para BAAI/bge-m3 (768 dims, maior qualidade multilíngue)

  ✅ Indexação no Qdrant
     — 562.152 vetores, distância cosseno
     — Payload por ponto: todos os metadados do chunk

  ✅ Indexação BM25
     — BM25Okapi sobre todos os 562.152 textos tokenizados
     — Serializado em pickle (bm25_index.pkl + bm25_ids.pkl)

  ✅ Busca Híbrida
     — Combina Qdrant (semântico) + BM25 (lexical)
     — Pesos: 0.6 semântico + 0.4 BM25
     — Expansão de contexto por chunks vizinhos
     — Filtros automáticos por número e ano do ato
     — Fallback sem filtro quando score < 0.5

  ✅ Query Optimization
     — Rewriting, HyDE, Decomposição, Filtros, Memória de conversa

  ✅ API FastAPI + Interface Streamlit
     — POST /query, GET /health, Swagger em /docs
     — Cache LRU (100 entradas, TTL 5min)
     — Histórico de conversa (últimos 6 turnos enviados)

4. O ESQUEMA DE BUSCA HÍBRIDA USA BM25 + VETORIAL COM RRF OU OUTRA ESTRATÉGIA?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  NÃO usa RRF (Reciprocal Rank Fusion).

  Usa WEIGHTED HYBRID SCORE:
    score_final = 0.6 × score_semântico + 0.4 × score_bm25

  Detalhes do pipeline de busca:
    1. Query entra no otimizador → gera hyde_texto, queries expandidas, filtros
    2. Embedding da query (ou do texto HyDE) via SentenceTransformer
    3. Qdrant: busca semântica com n_resultados × 10 candidatos (oversample)
    4. BM25: busca lexical com tokenização por tipo de pergunta
    5. Fallback: para payloads ausentes do BM25, busca payload no Qdrant por chunk_id
    6. Expansão: recupera chunks vizinhos (±2 chunks por chunk base, até 8 base)
    7. Fusão: combina scores semântico + BM25 com pesos configuráveis
    8. Rerank final por score_final, limitado a n_resultados
    9. Se score máximo < 0.5: refaz sem filtros de metadados (modo "menção")

CONTEXTO
━━━━━━━━
Este projeto foi desenvolvido por um grupo de estudos de NLP como desafio prático:
construir um sistema RAG funcional sobre documentos reais do setor regulatório
brasileiro. O sistema será avaliado por um especialista do setor elétrico usando
um benchmark de 45 perguntas em 10 categorias temáticas.

A ANEEL (Agência Nacional de Energia Elétrica) publica seus atos normativos em um
portal com proteção Cloudflare. Os dados cobrem 3 anos (2016, 2021, 2022) e incluem
diferentes tipos de ato: Resoluções Normativas (REN), Homologatórias (REH),
Autorizativas (REA), Despachos (DSP) e Portarias (PRT).

FASE 1 — ENGENHARIA DE DADOS (P1)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
O ponto de partida são 3 arquivos JSON brutos obtidos pelo scraping do portal da
ANEEL. Cada arquivo cobre um ano (2016, 2021, 2022) e contém metadados de todos
os atos normativos: número, tipo, data, ementa, situação, links para PDFs.

  limpar_json_aneel.py
    — Filtra atos vigentes (situação = "Em vigor")
    — Normaliza campos (tipo_codigo, data, numero)
    — Saída: aneel_vigentes_completo.json (~X atos vigentes)

  baixar_pdfs_aneel.py
    — Lê os links de cada ato e baixa os arquivos correspondentes
    — Usa curl_cffi com impersonação Chrome 124 para contornar o Cloudflare
    — Suporta: PDF, HTML (com seguimento de links), ZIP (extrai), XLSX, RAR
    — Download paralelo com workers configuráveis
    — Salva falhas em pdfs/falhas_download.json para reprocessamento
    — Estrutura: pdfs/ANO/CATEGORIA/ (texto_integral, voto, nota_tecnica, etc.)

  parser.py
    — Processa cada arquivo baixado e extrai texto estruturado
    — PDF digital: PyMuPDF para texto limpo, pdfplumber para tabelas
    — PDF escaneado: detectado como "pdf_imagem" (sem OCR)
    — HTML: BeautifulSoup4 extrai texto limpo
    — XLSX: openpyxl lê células por linha/coluna
    — Divide texto em chunks por parágrafo (800 chars, 120 overlap)
    — Filtra chunks menores que 50 chars
    — Salva incrementalmente em Parquet via PyArrow (streaming, ~111MB final)

  chunker_json.py
    — Processa as ementas do JSON (resumos dos atos)
    — Usa RecursiveCharacterTextSplitter do LangChain
    — Saída: chunks_json_todos.parquet (16.167 chunks, 2.4MB)

  unir_parquets.py
    — Concatena chunks de ementas + chunks de PDFs/HTMLs/XLSXs
    — Alinha colunas, resolve tipos, remove duplicatas por chunk_id
    — Saída: chunks_completo_unificado.parquet (562.152 chunks, ~111MB)

FASE 2 — BUSCA HÍBRIDA (P2)
━━━━━━━━━━━━━━━━━━━━━━━━━━━
  p2_indexar.py
    — Lê o parquet unificado
    — Gera embeddings locais com SentenceTransformer (MiniLM-L12-v2, 384 dims)
      em batches de 256 (CPU ou GPU se disponível)
    — Indexa no Qdrant (coleção "aneel_chunks", distância cosseno)
    — Constrói índice BM25Okapi sobre todos os textos tokenizados
    — Serializa o BM25 em pickle (db/bm25/bm25_index.pkl + bm25_ids.pkl)
    — Salva os BM25 IDs no payload Qdrant para recuperação cruzada

  Função buscar()
    — Parâmetros: query, qclient, bm25, bm25_ids, n_resultados, filtros,
                  hyde_texto, tipo_pergunta
    — Extrai filtros automáticos (numero/ano) da query por regex
    — Embedding: usa hyde_texto se disponível, senão a query direta
    — Qdrant: busca com oversample (n × 10), aplica filtros de metadados
    — BM25: tokeniza query, expande termos por tipo de pergunta,
            busca top-N por score TF-IDF
    — Recuperação de payload: chunks BM25 sem vetores buscados no Qdrant por chunk_id
    — Expansão de contexto: recupera ±2 chunks vizinhos por chunk base
    — Fusão: score_final = 0.6 × semântico + 0.4 × BM25
    — Fallback: se score máximo < 0.5, refaz sem filtros (modo menção)
    — Retorno: lista de dicts com texto, metadados e scores

FASE 3 — API, LLM E INTERFACE (P3)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  main.py (FastAPI)
    — Lifespan: inicializa QdrantClient, carrega BM25, instancia modelos
    — POST /query: recebe QueryRequest (pergunta + histórico de 6 turnos)
    — Verifica cache LRU (100 entradas, TTL 5min) antes de processar
    — ThreadPoolExecutor: paraleliza query_optimizer + busca
    — Timeout de 12s no otimizador (fallback para query original)
    — Roteamento: analítico → analytics.py, RAG → busca + LLM,
                  ambíguo → retorna pergunta de esclarecimento
    — Retorno: QueryResponse com resposta, fontes, tipo, cache_hit, latência

  query_optimizer.py
    — Entrada: pergunta + histórico de conversa + contador de esclarecimentos
    — Saída: QueryOtimizada (queries, hyde_texto, filtros, tipo_pergunta,
             ambigua, contexto_mudou, pergunta_esclarecimento)
    — Operações (via Groq/Llama, prompt único):
      · Metadata Extraction → número, ano e tipo_codigo do ato na pergunta
      · Query Rewriting → expande vocabulário técnico ANEEL
      · HyDE → gera trecho hipotético de resolução para embedding
      · Decomposição → divide em sub-queries para perguntas compostas
      · Resolução de referências → "esse despacho" → id do ato mencionado antes
      · Detecção de mudança de contexto → descarta histórico se necessário
      · Detecção de ambiguidade → solicita esclarecimento (máx 3x por contexto)
      · Mesclagem → adiciona respostas anteriores do mesmo contexto como contexto

  llm_chain.py
    — Suporte a dois provedores com fallback automático:
      · Groq (llama-3.3-70b-versatile) — principal, gratuito (100k tokens/dia)
      · DeepSeek R1 (deepseek-reasoner) — fallback quando Groq retorna 429
      · Integração via API OpenAI-compatível (pacote openai + base_url DeepSeek)
    — 6 system prompts especializados:
      · BUSCA    — perguntas factuais diretas
      · RESUMO   — sumarização de atos longos
      · COMPARACAO — diferenças entre atos/versões
      · TABELA   — dados estruturados de XLSXs
      · ESPECIFICA — artigos e cláusulas específicas
      · MENCAO   — documento não disponível na base (modo menção)
    — Anti-alucinação explícita: "responda APENAS com base nos documentos"
    — Instrução de recusa obrigatória quando não há informação suficiente
    — MAX_TOKENS = 2048

  analytics.py
    — Detecta perguntas analíticas ("quantas", "ranking", "mais frequente")
    — Opera sobre aneel_limpo_completo.json com pandas (sem chamar o LLM)
    — Suporta: contagem, ranking por mês/autor/tipo, listagem filtrada
    — Resposta instantânea, sem custo de tokens

  app.py (Streamlit)
    — Histórico de conversa em session_state (últimos 10 turnos)
    — Envia últimos 6 turnos com cada requisição à API
    — Contador de esclarecimentos por contexto (máx 3)
    — Botão "Limpar histórico" na sidebar
    — Exibe: resposta, fontes, tipo_resposta, cache_hit, scores semântico/BM25

RESULTADOS E MÉTRICAS
================================================================================

DADOS INDEXADOS
  562.152 pontos no Qdrant (vetores 384 dims)
  562.152 documentos no BM25
  Anos: 2016, 2021, 2022
  Tamanho médio de chunk: ~598 chars

PERFORMANCE (estimativa)
  Latência mediana de resposta: ~800ms a 2.000ms
  Query optimizer (Groq): ~300–500ms
  Geração de resposta (Groq): ~500–1.500ms
  Cache hit: ~0ms (LRU)

LIMITAÇÕES
  — Apenas 3 anos disponíveis (2016, 2021, 2022)
  — ~25.202 de ~27k arquivos baixados (restante bloqueado por Cloudflare)
  — XLSXs não têm cabeçalho por chunk (cálculos financeiros não funcionam)
  — 100k tokens/dia no Groq gratuito (~60–70 perguntas complexas/dia)
  — Embeddings em migração (MiniLM → BGE-M3)
